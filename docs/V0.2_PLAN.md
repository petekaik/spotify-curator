# Hermes Spotify Curator — Architecture Plan

**Date:** 2026-07-08
**Status:** Planning phase
**Target:** v0.2.0 — Hermes-integrated QNAP service

## 🎯 New vision

A **dedicated Hermes Agent instance** running on QNAP TS-251 that:
1. Analyzes the user's Spotify listening history on a schedule
2. Generates **mood/genre-themed "weekly recommendations"** playlists
3. Persists results to QNAP storage (not just Spotify)

The output playlists are categorized by **mood** (cheerful, calming, stimulating,
focus) and **genre** (indie, post-rock, ambient, etc.).

---

## What changes from current v0.1.0

| Aspect | v0.1.0 (current) | v0.2.0 (target) |
|---|---|---|
| **Deployment** | Local Mac CLI | QNAP container (24/7) |
| **Trigger** | Manual `spotify-curator` command | Scheduled cron + on-demand |
| **Output** | One playlist per run | Multiple mood/genre playlists |
| **Storage** | Spotify only | Spotify + QNAP Parquet/JSON reports |
| **LLM role** | None (rule-based ranking) | LLM mood classification + commentary |
| **User interface** | CLI | CLI + (optional) daily digest via TTS |
| **Authorization** | User runs OAuth locally | Pre-authorized headless service |

## Three new FPs

### FP-3b/c/d — Additional discovery sources

Bring in three more data sources for richer candidate pools.

#### FP-3b: MusicBrainz
- **Why:** Spotify removed genre data, Last.fm tags are sparse. MusicBrainz has
  the best genre taxonomy and is free, no auth.
- **Implementation:**
  - `src/discovery/sources/musicbrainz.py` — wrapper using `musicbrainzngs`
  - `mbid_get_artist(mbid)` — get full artist data + genres
  - `search_artists_by_genre(genre, limit=50)` — discovery by tag
  - Rate limit: 1 req/sec (be polite)
- **Tests:** ~12 unit tests
- **Dependencies:** `musicbrainzngs` (already in requirements)

#### FP-3c: Bandcamp scraping
- **Why:** Post-rock, ambient, drone, experimental, lo-fi — these genres
  publish to Bandcamp FIRST, before Spotify. Indie/small-label community.
- **Implementation:**
  - `src/discovery/sources/bandcamp.py` — HTML scraper using httpx + selectolax
  - `get_tag_artists(tag, sort='top', limit=30)` — parse `bandcamp.com/tag/<tag>`
  - `get_artist_followers(artist_url)` — extract from artist page
  - Identify emerging: < 1000 followers
  - Rate limit: 1 req/2s
- **Tests:** ~10 unit tests with vcrpy cassettes

#### FP-3d: Reddit community mining
- **Why:** Real humans sharing, voting, surfacing artists. "FRESH" threads
  in r/indieheads, r/postrock, etc. are weekly goldmines.
- **Implementation:**
  - `src/discovery/sources/reddit.py` — uses public JSON API (no auth)
  - `get_fresh_thread_artists(subreddit, time_filter='week')` — parses
    weekly "FRESH" / "What are you listening to" threads
  - Regex artist name → resolve via MusicBrainz or Spotify search
  - Score by mention count + upvote ratio
- **Tests:** ~10 unit tests

### FP-7 — Integration tests with vcrpy

Real end-to-end tests that record actual API responses and replay them
deterministically. **Critical for CI** (no network in CI).

- **Implementation:**
  - `tests/integration/` — separate test directory
  - `tests/fixtures/cassettes/` — recorded HTTP responses
  - Mark all integration tests with `@pytest.mark.integration`
  - Provide a way to **record** new cassettes: `pytest --record-mode=once`
  - For Spotify: spotipy's internal HTTP layer can be intercepted
  - For Last.fm / ListenBrainz / MusicBrainz / Bandcamp / Reddit: vcrpy
- **Coverage:**
  - Full `playlist create` flow with recorded Spotify + Last.fm responses
  - Profile build with recorded top-artists responses
  - Discovery chain: Last.fm + ListenBrainz cross-reference
  - Edge cases: 4xx errors, rate limits, timeouts
- **CI:** GitHub Actions uses recorded cassettes (no live API calls)

#### Mood classification

Use a small LLM (or rule-based for v0.2.0) to classify artists into moods.

**Mood taxonomy (v0.2.0):**
- **cheerful** — uplifting, energetic, positive
- **calming** — peaceful, ambient, slow
- **stimulating** — complex, intellectual, challenging
- **focus** — minimal, instrumental, work-friendly
- **melancholic** — sad, reflective, emotional
- **energetic** — high-tempo, driving, intense

**v0.2.0 strategy (rule-based, no LLM yet):**
- Use Spotify audio features (deprecated but still working)
- Heuristics: tempo, valence, energy, acousticness
- Map to mood via thresholds

**v0.3.0+ (LLM-enhanced):**
- Use small local LLM (Ollama on QNAP) for richer classification
- Generate weekly commentary text explaining the picks
- TTS playback via existing Hermes TTS chain

#### IPC interface

The container exposes a simple HTTP server (stdlib `http.server`, no extra deps):

```
GET  /health
GET  /status
POST /generate            # full weekly run
POST /generate?mood=calming # one-mood run
POST /profile/refresh     # rebuild profile from Spotify
GET  /reports             # list past runs
GET  /reports/<id>        # show specific report
```

This lets the user trigger runs from a phone, from another Hermes instance,
or from a future web UI — without SSH into QNAP.

#### Per-mood playlist flow

For each mood in taxonomy:
1. Query discovery sources (Last.fm, ListenBrainz, MusicBrainz) for candidates
2. Filter candidates by genre overlap with user's top artists in that mood
3. Apply ranking algorithm with mood-specific weight overrides
4. Take top N=10 artists, fetch 2-3 tracks each
5. Build a playlist named e.g. "Weekly Cheerful — YYYY-Wnn"
6. Write to Spotify
7. Persist report to QNAP storage (Parquet + JSON metadata)

#### Scheduled job (cron)

Inside the container, a lightweight scheduler:
- Sundays 23:00 — full weekly digest
- First of month 23:00 — refresh profile + re-evaluate
- Handled by `cron` inside the container, or Python's `schedule` library

The scheduler is **self-restarting** if the container is killed and respawned.

#### Storage layout on QNAP

```
/share/homes/<user>/.cache/spotify-curator/
├── profile.parquet
├── discovery/                          # last 7 days of results
│   ├── 2026-07-08_lastfm_indie_rock.json
│   ├── 2026-07-08_listenbrainz.json
│   └── ...
└── /share/CACHEDEV1_DATA/spc-reports/
    ├── 2026-W27/
    │   ├── weekly_digest.md
    │   ├── playlists.json               # links to Spotify playlists
    │   ├── artists_picked.parquet
    │   └── commentary.txt               # optional, LLM-generated
    └── 2026-W28/...
```

## Hermes Skill + Tools

Per the new direction ("tehdään tästä skill + tools Hermes agentille"),
this isn't just a CLI anymore — it becomes a **Hermes skill** that other
Hermes instances can invoke.

### Skill structure

```
skills/spotify-curator/
├── SKILL.md                          # how to invoke
├── references/
│   ├── commands.md                   # full CLI reference
│   ├── api.md                        # IPC API reference
│   └── mood_taxonomy.md              # mood definitions
└── scripts/
    ├── generate_weekly.sh            # one-shot CLI
    ├── query_status.sh
    └── trigger_mood.sh
```

### Tools exposed to other Hermes agents

When installed as a Hermes skill, exposes these tools:

```
spotify_curator_generate_weekly       # full weekly run
spotify_curator_generate_mood         # single mood (mood: str)
spotify_curator_get_status            # health + last run info
spotify_curator_get_recent_reports    # list past N runs
spotify_curator_get_report            # get specific report by ID
spotify_curator_refresh_profile       # rebuild profile
```

Each tool returns a structured dict (JSON-serializable) for other agents.

### Other Hermes instances can use it like:

```python
# From any Hermes session:
result = await tool("spotify_curator_generate_mood", mood="calming")
# Returns: {"playlist_url": "...", "tracks": [...], "artists": [...]}

# Use the result to drive a TTS summary:
await tool("tts_speak", text=f"This week's calming picks: {result['summary']}")
```

## Effort estimate (FPs 3b/c/d, 7, 9)

| FP | Components | Effort | Notes |
|---|---|---|---|
| **FP-3b** MusicBrainz | 1 source + tests | 1-2 days | Already in requirements |
| **FP-3c** Bandcamp | 1 source + tests | 2-3 days | HTML scraping, edge cases |
| **FP-3d** Reddit | 1 source + tests | 1-2 days | JSON API is simple |
| **FP-7** Integration tests | 4 test files + cassettes | 2-3 days | Mostly recording & verifying |
| **FP-9** Daemon + mood | daemon/ + hermes_skill/ + deploy/ | 5-7 days | Largest new feature |

**Total:** ~12-17 days of solid work, or ~3 weeks with breaks.

**MVP suggestion:** Do FP-3b/c/d first (small, contained), then FP-9 mood
classification with the existing discovery sources (skip Reddit/Bandcamp for
the first daemon release), then FP-7 tests, then the full QNAP deployment.

## Open questions for the user

1. **Mood taxonomy:** Is the 6-mood list (cheerful, calming, stimulating,
   focus, melancholic, energetic) right? Or do you want a different
   categorization?
2. **Schedule:** Weekly (Sunday 23:00) is the default. Would you prefer
   different day/time?
3. **Storage format on QNAP:** Parquet for structured data, Markdown for
   human-readable digest. OK?
4. **Hermes skill location:** Should the skill be in
   `~/.hermes/skills/spotify-curator/` on the **Mac** (for dev/testing) or
   `~/.hermes/skills/spotify-curator/` inside the **QNAP container**?
5. **LLM commentary:** v0.2.0 uses rule-based mood classification. Add LLM-
   generated commentary in v0.3.0? If yes, which LLM (Ollama? OpenAI? local
   Phi-3?)
6. **TTS digest:** Should the weekly report also be playable via the
   existing Hermes TTS chain (we already verified this works — see memory)?
